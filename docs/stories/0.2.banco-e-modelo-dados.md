## Story
**As a** lojista,
**I want** o banco de dados modelado com produtos configuráveis (JSONB) e pedidos imutáveis,
**so that** novos produtos com parâmetros diferentes não exijam código novo e nenhum pedido perca sua especificação.

## Acceptance Criteria
1. PostgreSQL provisionado (provedor decidido na story 0.1 AC-7)
2. Tabela `products`: id, slug (unique), name, description, base_price (centavos, int), status (draft/published), photos, `variants` JSONB `[{label, model_url, price_delta, dimensions}]`, `param_schema` JSONB conforme PRD §8
3. Tabela `orders`: id, created_at, status (`pending/paid/production/shipped/delivered`), stripe_session_id, total_amount (centavos), `customer` JSONB {name, email, phone, address}, product_id (FK), `configuration` JSONB imutável, snapshot_url
4. Migrations versionadas e reproduzíveis do zero (`migrate reset` funciona)
5. Seed do comedouro pet com o `param_schema` exato do PRD §8 (pet_name/color/size)
6. Camada de acesso tipada (ORM decidido na 0.1) com tipos TypeScript para `param_schema` e `configuration`

## Dev Notes
- Valores monetários SEMPRE em centavos (int) — evita float e casa com a Stripe.
- `configuration` do pedido é imutável por regra de negócio (P-03): não criar endpoint de update; correções via reprocessamento (G-08, fase 2).
- O `param_schema` é a fonte única de verdade compartilhada entre visualizador (Épico 1), preço (2.3) e gerador 3MF (Épico 5) — modelar pensando nos três consumidores. Exemplo canônico no PRD §8.
